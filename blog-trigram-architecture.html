React + PixiJS Architecture: Browser-Based Combat Simulation

Web-based combat simulation = democratized access. Documented in ARCHITECTURE.md, Black Trigram runs entirely in-browser using React 19 + PixiJS 8 + TypeScript. Zero backend. Zero installation. Zero platform lock-in. Open browser ‚Üí train martial arts. This isn't compromise for convenience‚Äîit's intentional accessibility architecture. Fighting games historically gatekept behind console hardware and local multiplayer. We chose universal access instead.

1. üñ•Ô∏è React UI Layer

Korean-themed components with responsive design. Screens: CombatScreen, TrainingScreen, IntroScreen. Common UI: CombatHUD, TrigramWheel, ProgressTracker. Base modules: BaseButton, KoreanText, BackgroundGrid. Functional components with TypeScript throughout.

2. ‚öôÔ∏è Game Logic Engine

TypeScript modules under src/systems/: CombatSystem orchestrates input ‚Üí trigram ‚Üí vital-point ‚Üí damage ‚Üí feedback. TrigramSystem manages stances via StanceManager, TransitionCalculator, TrigramCalculator, KoreanCulture. VitalPointSystem handles 70 vital points through KoreanAnatomy, HitDetection, DamageCalculator.

3. üé® PixiJS Renderer

PixiJS 8 with WebGL for 60fps 2D graphics. Manages character sprites (PlayerVisuals, EnemyVisuals), dojang background, particle systems (HitEffectsLayer), UI overlays. @pixi/react integration wraps PixiJS Application in React components.

4. üéµ Audio Engine

Howler.js + Web Audio API for Korean traditional music blended with cyberpunk SFX. AudioManager loads bone cracks, impacts, ambient sounds. VariantSelector randomizes hit sounds. DefaultSoundGenerator provides fallback audio.

5. üóÑÔ∏è State Management

Zustand + React Context for in-browser state. useGameState, useUIState, useEnemyState track health, stamina, Ki, stance, combat metrics. No persistence‚Äîrefresh resets progress. Client-side only, no backend dependencies.

Full architecture documentation: ARCHITECTURE.md includes C4 Container diagrams, Component views, Combat Flow sequences. TypeScript provides type safety across all layers. Vite handles build optimization with tree-shaking.

Web applications without layer enforcement become unmaintainable. React for UI, PixiJS for rendering, TypeScript for logic‚Äîclear separation enables confident refactoring without breaking systems.

ü•ä Combat System: Making 70 Vital Points Actually Work

Hit Detection Reality: Simon describes 70 vital points elegantly. I implemented hitbox-to-hurtbox collision that runs 120 times per second. Each fighter: 23 body parts (synchronicity!), each part has collision polygon. Attack hitbox sweeps through space during animation frames. Separating Axis Theorem (SAT) for polygon collision detection. When hitbox overlaps hurtbox ‚Üí check which vital point ‚Üí calculate damage multiplier (1.0√ó to 5.0√ó depending on location).

Damage Calculation: Base damage √ó vital point multiplier √ó stance modifier √ó balance factor √ó random variance (¬±5%). Six combat stats interact: Health, Pain, Balance, Consciousness, BloodLoss, Stamina. Each affects others‚Äîhigh pain reduces balance, low stamina amplifies damage, blood loss drains consciousness. Simulating realistic combat degradation in JavaScript. Code at src/systems/VitalPointSystem/DamageCalculator.ts.

The 70 Vital Points: Implemented as JSON data structure‚Äîeach point has coordinates (x,y), severity (1-5), effect type (stun/knockback/bleed), and Korean name. Head: 10 points. Torso: 20 points. Arms: 16 points. Legs: 16 points. Neck: 8 points. Total: 70. Visualized during debug mode‚Äîdraw circles over each vital point location. Players discovering head strikes do 5√ó damage = intended design working.

Second Panic Moment: Vital point at (x: 0.5, y: 0.7) = neck. Attack landed at (x: 0.5, y: 0.69). Collision missed by 0.01 units. Players complaining hits not registering. Solution: 5-pixel hitbox expansion tolerance. Real physics is unforgiving‚Äîgame physics needs generous collision boxes. Combat feel > realistic collision precision.

‚ö° Performance: 60fps or Bust

Optimization Reality: Browser games = performance constraints. Target: 60fps on 5-year-old laptops (Law of Fives demanding accessibility). Profiled with Chrome DevTools. Bottlenecks discovered: collision detection (38% frame time), PixiJS rendering (23% frame time), audio processing (18% frame time), state updates (16% frame time), misc (5% frame time). Percentages adding to 100% = good profiling; percentages summing to 23% somewhere = synchronicity.

Performance Wins:

  • Spatial hashing for collision: Only test nearby hitboxes. Reduced collision checks from O(n¬≤) to O(n). 5√ó speedup.
  • Object pooling for particle effects: Reuse particle objects instead of create/destroy. Garbage collection stutter eliminated. Consistent 60fps even during 5-hit combos.
  • Texture atlases: Combined 50+ sprite images into 5 texture atlases (Law of Fives manifesting in asset pipeline). GPU draw calls reduced from 50 to 5. 40ms ‚Üí 8ms render time.
  • Web Workers for AI: Enemy AI calculations in separate thread. Main thread free for rendering. Frame drops eliminated during complex AI decision-making.
  • Lazy loading audio: Load SFX on-demand, not at startup. Initial load time: 23 seconds ‚Üí 5 seconds. Golden ratio improvement (23% of original).

Third Panic Moment: iOS Safari performance 10√ó worse than Chrome. WebGL context limits, garbage collection aggressive, requestAnimationFrame throttled. Solution: Reduced particle count on mobile detection, disabled shadows, lowered texture resolution. Performance vs. visual fidelity trade-off = pragmatic compromise. Working game on iPhone > beautiful slideshow.

üé® PixiJS Rendering: Making Pixels Fight

Render Pipeline: PixiJS 8 with WebGL backend. Five rendering stages: Background (dojang grid), Characters (sprite animations), Effects (particles/blood), UI (health bars/combos), Debug (hitboxes/collision). Layer ordering critical‚ÄîUI must render last, debug must render on top. Z-index management via PixiJS container nesting. Each container = PixiJS DisplayObject tree. Total sprites per frame: ~150. GPU handles it easily (modern GPUs laugh at 150 sprites).

Animation System: Sprite sheets generated from Aseprite. Each fighter: 23 animations (idle, walk, 5 attacks, 3 blocks, dodge, hit reaction, knockdown, victory, defeat, stances). Frame data: attack active frames 3-7, recovery frames 8-23 (synchronicity in frame counts). Animation state machine transitions based on input + combat state. Implementation: src/rendering/PlayerVisuals.ts.

Particle Effects: Blood splatter, hit sparks, dust clouds. Each particle: velocity, lifetime, fade, color. Particle emitter spawns 5-23 particles per hit (range depending on damage). Particles update every frame, die after 0.5-2.3 seconds (golden ratio timing). WebGL batch rendering keeps particles cheap‚Äî500 particles = 2ms render time.

Fourth Panic Moment: Blood particle effects causing memory leak. 10 minutes of combat = 2GB RAM usage = browser crash. Issue: Particles not being destroyed after lifetime expired. Fixed by proper object cleanup in particle system. Memory stable at 150MB now. Lesson: JavaScript garbage collection is not magic‚Äîclean up your references.

üîÆ Future Development: Unity Migration (Maybe)

Current Limitation: Browser-based = no native performance, no console ports, no VR support. Future plan: Migrate to Unity while keeping React for menus. Why Unity Eventually:

  • Physics engine: Unity's built-in physics > JavaScript approximations. Realistic ragdoll, collision layers, joints.
  • Platform support: Unity exports to 23+ platforms. Web version serves 5 platforms max (Windows/Mac/Linux/iOS/Android browsers).
  • VR capabilities: Unity has mature VR SDKs. WebXR exists but immature. First-person martial arts VR = killer feature requiring Unity.
  • Asset pipeline: Unity's asset importers, LOD generation, shader management superior to web tooling.
  • Performance ceiling: Native C++ rendering > JavaScript WebGL. 60fps web becomes 144fps native. Competitive play demands performance.

Current Status: Web version = MVP. Proves gameplay concepts, validates combat system, builds community. Unity migration = expensive. Justify cost with proven demand. Pragmatic development over premature optimization. Current game playable, fun, educational. Unity version = enhancement, not necessity.

Architectural Wisdom: Start simple (web), prove concept, migrate to complex (Unity) only when justified. Many games fail by over-engineering first version. We shipped playable game in 6 months. Unity port would've taken 2 years. Speed to player feedback > architectural purity.
