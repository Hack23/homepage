The Five Practices of Not Shipping Vulns

1. Threat Model First

Design with adversaries in mind. What can go wrong? What are we doing about it? Answer before writing code, not after.

Code written without threat modeling is just bug creation with optimism.

2. Secure by Default

Default configurations should be secure. Users shouldn't have to know what flags make things safe. Safe should be the default.

If secure configuration requires reading docs, most deployments are insecure.

3. Least Privilege

Every process, every user, every component‚Äîminimum necessary permissions. Privilege escalation is harder when you start with nothing.

Root by default is convenience for attackers, not users.

4. Input Validation

Trust nothing from users, APIs, databases, files. Validate, sanitize, escape. SQL injection isn't clever‚Äîit's inexcusable.

All input is hostile until proven otherwise. User input is especially hostile.

5. Output Encoding

XSS happens because output isn't encoded. Context-aware escaping. HTML entity encoding. It's not hard‚Äîit's discipline.

XSS in 2025 means you didn't use your framework's built-in protections.

Secure SDLC: Security at Every Stage

Security integrated throughout development, not bolted on at the end:

  • Design ‚Äî Threat modeling, security requirements, architecture review
  • Development ‚Äî Secure coding standards, linters, IDE security plugins
  • Testing ‚Äî Unit tests for security properties, SAST, DAST, dependency scanning
  • Review ‚Äî Security-focused code review, pen testing, bug bounties
  • Deployment ‚Äî Secure configuration, secrets management, least privilege
  • Operations ‚Äî Monitoring, logging, incident response, patching

META-ILLUMINATION: Security after deployment is incident response. Security during development is prevention.

Tools: Automate What You Can

Humans make mistakes. Computers don't get tired. Automate security checks:

  • Static Analysis (SAST) ‚Äî SonarQube, CodeQL, Semgrep. Find vulns in source code before runtime.
  • Dynamic Analysis (DAST) ‚Äî OWASP ZAP, Burp Suite. Test running applications for vulns.
  • Dependency Scanning ‚Äî Dependabot, Snyk, OWASP Dependency-Check. Find known vulns in libraries.
  • Secret Scanning ‚Äî GitGuardian, TruffleHog. Prevent credentials in repos.
  • Container Scanning ‚Äî Trivy, Clair. Scan images for vulns before deployment.

Tools aren't perfect, but they catch the obvious stuff humans miss at 2am.

CHAOS ILLUMINATION: Automated tools find known vulns. Code review finds design flaws. You need both.

Dependency Hell: Trust You Can't Verify

Modern applications are mostly dependencies. Every dependency is trust you can't fully verify:

  • Vet Dependencies ‚Äî Check maintainer reputation, update frequency, known vulns
  • Pin Versions ‚Äî Exact versions, not ranges. Reproducible builds.
  • Monitor for Vulns ‚Äî Automated scanning, CVE monitoring, patch fast
  • Minimize Dependencies ‚Äî Every library is attack surface. Do you really need it?
  • Supply Chain Security ‚Äî SBOM, signed artifacts, provenance tracking

Operation Mindfuck the supply chain attackers: Minimize dependencies, pin versions, scan everything, patch fast.

ULTIMATE ILLUMINATION: The most secure code is code you didn't write. The most vulnerable code is code someone else wrote that you blindly trust.
