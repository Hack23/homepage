Question: How do you secure a system designed for radical transparency?

Answer: Make the security architecture transparent too. Not security through obscurity‚Äîsecurity through mathematical proof and public evidence.

When I architected security for the Citizen Intelligence Agency, conventional wisdom said: "Don't publish threat models. Don't document attack surfaces. Don't reveal defensive strategies." Conventional wisdom creates security theater‚Äîconfidence without evidence.

Think for yourself. We publish everything: current security architecture, future security roadmap, complete STRIDE threat model, and CRA compliance assessment. Everything. Public. Verifiable.

Illumination: Security through obscurity fails when attackers have time. Security through mathematical proof holds when attackers have eternity. Choose accordingly.

The Five Defensive Layers

Defense-in-depth organized by the Law of Fives. Each layer independent. Breach one, four remain. Classic military strategy meeting modern cybersecurity.

1. üåê Perimeter Defense: The Outer Pentacle

CloudFlare DDoS protection. 5 Gbps baseline, elastic to 5 Tbps. Not because we expect Tbps attacks‚Äîbecause planning for pentagonal scaling (5^3 = 125 growth factor) provides comfortable margins.

WAF rules in five categories: SQL injection patterns, XSS attempts, path traversal, credential stuffing, API abuse. Each category tuned independently. Alert on 5 failed attempts in 5 minutes‚Äîsynchronistic thresholds that balance security with usability.

Geographic distribution: CloudFront edge locations across 5 continents. Europe (primary), North America, Asia, South America, Oceania. Africa planned for year 5 of roadmap‚Äîcompleting the global pentacle.

Perimeter defense that blocks legitimate users fails its mission. Perimeter defense that lets attackers through fails harder. Balance through the golden ratio: 61.8% security strictness, 38.2% user convenience.

2. üö™ Application Gateway: Identity Verification

Spring Security managing authentication. Five auth levels: anonymous (read-only public data), authenticated (vote tracking), verified (bookmarks/notes), analyst (advanced queries), admin (system management).

Session management: JWT tokens valid for 23 minutes. Not arbitrary‚Äîtesting showed 20 minutes = frequent re-auth annoyance, 25 minutes = security window too wide, 23 = sweet spot. The number revealing itself through user behavior analysis.

OAuth2 integration with five providers: GitHub, Google, Microsoft, LinkedIn, BankID (Swedish e-identity). Users choose their trust anchor. We don't mandate.

Rate limiting: 5 requests/second per anonymous IP, 23 requests/second per authenticated user, 115 requests/second for verified analysts (5 √ó 23). Synchronistic thresholds that emerged from production load patterns.

Authentication without authorization is security theater. Authorization without authentication is chaos. Both required. Both enforced. Both audited.

3. üõ°Ô∏è Business Logic Validation: The Sacred Rules

Input validation at five checkpoints: syntactic (well-formed?), semantic (meaningful?), business (allowed?), temporal (current?), referential (consistent?).

Example - politician vote recording:

  • Syntactic: Vote ID matches UUID pattern
  • Semantic: Vote value in {yes, no, abstain, absent, refrain}
  • Business: Politician active during vote date
  • Temporal: Vote date not future, not before parliament formation
  • Referential: Vote links to existing proposal and session

Fail any checkpoint = rejection. Pass all five = acceptance. Boolean AND across five gates. Simple. Effective. Auditable.

STRIDE threat mitigations: Spoofing (JWT signatures), Tampering (checksums), Repudiation (audit logs), Information Disclosure (encryption), Denial of Service (rate limits), Elevation of Privilege (least-privilege model). Six STRIDE categories mapped to five defensive layers via overlapping concerns‚ÄîDoS handled primarily at perimeter, secondarily at gateway.

Business logic vulnerabilities are the hardest to detect with automated tools. Why? They're not bugs‚Äîthey're features misused. Defense requires understanding the problem domain, not just running scanners.

4. üîí Data Protection: Encryption at Rest

PostgreSQL with transparent data encryption. Five encryption scopes: personal identifiers (emails, IPs), authentication credentials (hashed, salted), user-generated content (notes, bookmarks), cached query results (session-specific), backup archives (AES-256).

Key rotation schedule: Every 23 days for session keys, every 115 days (5 √ó 23) for database encryption keys, annually for backup encryption keys. Fibonacci-adjacent intervals balancing security refresh with operational stability.

Data classification: Public (parliament votes - 99% of data), Internal (user preferences), Confidential (authentication tokens), Restricted (admin keys), Secret (encryption master keys). Five levels matching our architectural layers.

GDPR compliance: Right to access, right to erasure, right to portability, right to rectification, right to restrict processing. Five rights, five implementation strategies, documented in SECURITY_ARCHITECTURE.md.

Encryption that can't be decrypted when needed is expensive data destruction. Encryption that decrypts too easily is expensive obfuscation. Balance through key management discipline.

5. üëÅÔ∏è Monitoring & Alerting: The All-Seeing Eye

Five observation channels: Application logs (what), System metrics (how much), Distributed traces (where), Security events (who/when), Audit records (compliance).

Log aggregation: Elasticsearch cluster with 5-node minimum. Primary + 4 replicas. Data retention: 23 days hot storage, 115 days warm storage (compressed), 2 years cold storage (archived). Retention periods matching attack investigation windows.

Alert thresholds: P1 (immediate) - authentication system down, data breach detected. P2 (5 minutes) - elevated error rates, suspicious access patterns. P3 (30 minutes) - capacity warnings, configuration drift. P4 (2 hours) - performance degradation. P5 (daily) - trend analysis, vulnerability scanning results.

Incident response: Detection (automated), Analysis (5-minute target), Containment (15-minute target), Eradication (1-hour target), Recovery (4-hour target), Lessons Learned (documented within 5 days). The five-phase cycle documented in SECURITY_ARCHITECTURE.md.

Monitoring that doesn't inform action is surveillance without purpose. Monitoring that enables rapid response is defensive intelligence. Know the difference.

STRIDE Threat Model: The 23 Attack Scenarios

Read the full threat model: THREAT_MODEL.md contains attack trees, probability assessments, impact analysis, and mitigations for all 23 scenarios.

The five highest-impact scenarios:

üéØ Scenario 5: SQL Injection via Search

Attack: Malicious SQL in politician name search. Mitigation: Parameterized queries (100% coverage enforced by static analysis), input sanitization, least-privilege database account. Probability after mitigation: 0.01% (requires five simultaneous control failures).

üîì Scenario 11: Session Hijacking

Attack: Steal JWT token, impersonate user. Mitigation: Short session lifetime (23 min), secure cookie flags, IP+User-Agent binding, token refresh rotation. Probability after mitigation: 0.5% (requires network-level attack window under 23 minutes).

‚ö° Scenario 15: DDoS Amplification

Attack: Abuse expensive queries to overwhelm database. Mitigation: Query complexity limits, response caching, rate limiting, CDN absorption, auto-scaling. Probability after mitigation: 2% (requires distributed coordination across 5+ attack sources).

üîë Scenario 19: Privilege Escalation

Attack: Modify request to gain admin access. Mitigation: Role-based access control (immutable after auth), server-side enforcement, audit logging, privilege verification at five checkpoints. Probability after mitigation: 0.1% (requires defeating multi-layer authorization).

üóÇÔ∏è Scenario 23: Data Poisoning

Attack: Submit fake parliament data. Mitigation: Source verification (Riksdag API cryptographic signatures), data consistency checks, anomaly detection, manual review queue for outliers. Probability after mitigation: 1% (requires compromising upstream Riksdag systems).

Threat modeling that stops at "here are the threats" is threat identification, not threat modeling. Real threat modeling includes: probability, impact, cost of mitigation, residual risk. We document all four dimensions.

Supply Chain Security: The Five Verification Layers

Open source project = hundreds of dependencies. Each dependency is a trust decision. We verify trust at five levels:

  1. FOSSA license scanning: Automated dependency analysis. Only approved open-source licenses. Alert on copyleft violations, commercial licenses, unknown licenses. Public badge shows real-time compliance.
  2. Snyk vulnerability scanning: Daily checks for CVEs in dependencies. Auto-PR for security updates. SLA: Critical CVE patches within 24 hours. High severity within 5 days. Medium within 23 days. The numbers again.
  3. Dependabot version monitoring: GitHub-native dependency updates. Reviews 5 outdated dependencies per week (prioritized by risk √ó usage). Systematic rather than reactive.
  4. SLSA Level 3 build provenance: Cryptographic attestation that artifacts came from our source. Tamper-proof supply chain. View our attestations‚Äîpublic evidence of build integrity.
  5. OpenSSF Scorecard continuous validation: Score 7.2/10 measuring branch protection, dependency updates, token permissions, code review, SAST integration, vulnerability disclosure. Automated evidence of security hygiene.

The 23rd dependency insight: When we inventory dependencies, critical path always includes 23-25 packages (varies by release). Why? Pareto principle in action‚Äî20% of dependencies (23 out of ~120 total) provide 80% of functionality. The natural clustering revealing itself.

Supply chain security that trusts by default is naive. Supply chain security that trusts nothing is paralyzed. We trust with verification‚Äîcryptographic, automated, continuous.

Security Wisdom: What the Patterns Taught Me

After architecting security for a transparency platform, what did I learn?

  1. Transparency is security's ally, not enemy. Publishing our threat model didn't help attackers‚Äîit helped defenders. Security researchers review our mitigations and suggest improvements. Crowdsourced security through radical openness.
  2. The 23 Enigma manifests in unexpected places. Session timeouts, rate limits, critical attack scenarios, key dependencies‚Äî23 keeps appearing. Not numerological superstition. Pattern recognition in action. When the same number clusters, investigate underlying structure.
  3. Defense-in-depth requires five independent layers minimum. Three layers = fragile. Four = asymmetric. Five = resilient. Pentagon structure resists single-point failures. One layer breached, four remain defending.
  4. Automated evidence > manual assurance. OpenSSF Scorecard, SLSA attestations, FOSSA scanning‚Äîcontinuous automated validation. Can't fake a cryptographic signature. Can't game a public transparency report. Mathematical proof > marketing claims.
  5. Security debt accumulates faster than technical debt. Skipped security review = future vulnerability. Delayed dependency update = time-bomb CVE. Disabled audit logging = forensic blindness. Pay security debt immediately or pay compound interest in breaches.

The map is not the territory, but our security map accurately describes our security territory because we update the map every time the territory changes. Documentation drift = security gap.
